# Nak-AI 技術設計書

本ドキュメントでは、Nak-AIの具体的な技術設計、アーキテクチャ方針、および将来的なUI移行を見据えた「設計のコツ（疎結合設計）」についてまとめます。

---

## 1. コア設計思想：処理エンジンとUIの完全な疎結合（疎結合設計のコツ）

Nak-AIは、プロトタイプ期の **「独立型デスクトップアプリ（Tkinter）」** から、将来の **「OBSカスタムブラウザドック（Webベース）」** へのフロントエンド移行を前提として開発します。
手戻りをゼロにし、最速で安定した開発を行うため、以下の**「疎結合設計」**を徹底します。

```
+-------------------------------------------------------+
|  【処理エンジン部】 (NakAiEngine: 純粋なPythonモジュール)  |
|  - マイク音声キャプチャ / 状態監視                     |
|  - Faster-Whisper によるローカルSTT                   |
|  - Gemini API による文脈分析・ヒント生成              |
|  - SQLite によるリスナー履歴 (RAG) 永続化             |
+-------------------------------------------------------+
                           │ (Queue / イベントコールバック)
                           ▼
+-------------------------------------------------------+
|  【UI表示部】 (フロントエンド / ガワ)                    |
|                                                       |
|   ● プロトタイプ期: Tkinter (Python標準GUI)            |
|     - 設定画面からマイクとAPIキーをエンジンに流す      |
|     - エンジンから届いた支援テキストを画面に表示       |
|                                                       |
|   ● 製品リリース期: カスタムブラウザドック (HTML/JS)     |
|     - エンジンを FastAPI 等の軽量Webサーバーに接続      |
|     - OBSのドック上に美しく描画                       |
+-------------------------------------------------------+
```

### 疎結合にするための実装ルール
* **UI依存コードの排除**:
  - 処理ロジックを担当するクラス（例: `NakAiEngine`）は、Tkinter などのGUIライブラリを一切インポートせず、ピュアなPythonコードとして構築します。
  - UIへのテキスト受け渡しは、スレッドセーフな `Queue` またはリスナー型のコールバック関数を介して行います。
* **GUIは単なる「表示器」にする**:
  - Tkinterのコードは、画面の描画と、ユーザー操作（ボタンクリック、設定入力）のイベントをエンジンに通知するだけの「ガワ」に徹します。複雑なロジックをGUIクラス内に記述してはいけません。
* **この設計によるメリット**:
  - **手戻りゼロの移行**: ステップ8でブラウザドック（HTML）に移行する際、Tkinterのコードを捨ててWebサーバーのラッパーを被せるだけで移行が完了します（コアの処理エンジンは1行も書き換える必要がありません）。
  - **テストの容易さ**: GUIを起動することなく、コマンドラインのテストスクリプトからエンジン単体を呼び出して、音声認識やLLMの挙動を自動テストできます。

## 2. 技術設計・実装方針

### ① アプリケーション形態
* **プロトタイプ期**: Python 3 と標準GUIライブラリ `Tkinter` で構築された、セットアップが容易で軽量なローカルデスクトップアプリケーション。
* **将来的な展開**: バックエンド（Python/Rust）とフロントエンドを分離し、OBSの「カスタムブラウザドック」に直接組み込み可能なタスクトレイ常駐型ハイブリッドアプリ（スタートアップ自動起動）へ移行します。

### ② 音声キャプチャ & ローカルSTT
* **音声入力の指定**: Discordなどの通話アプリと同様に、アプリの設定画面からキャプチャ対象のマイクデバイスを明示的に選択します（ゲーム音や他メンバーの通話音の混入を防ぎ、自分の声だけをクリアに拾うため）。
* **音声認識（STT）**: ローカルPC上で動作する **`Faster-Whisper`** を採用。外部サーバーに配信者の声を送信することなく、完全無料で高速・安全にテキスト化します。

### ③ コメント連携（Twitch特化）
* **対象プラットフォーム**: 初期開発フェーズでは **Twitch** に特化します。
* **接続方式**: ユーザーに複雑なOAuth認証を求めない、**「匿名IRC（Internet Relay Chat）プロトコル」**を利用してチャットをリアルタイムに受信します。

### ④ 無音検出（沈黙検知）とLLM送信タイミング（※本製品の工夫ポイント）
* 実況が途切れたことの判定は、マイク入力の音量レベル（RMS）監視とSTTの出力インターバルを組み合わせたハイブリッド方式を想定しています。
* まず基本となるSTTのリアルタイム処理を実現した後に、どのタイミングでLLMにリクエストを投げるかを含め、アルゴリズムのチューニングを詳細に行います。

### ⑤ クラウドLLM & セキュアな認証
* **LLMモデル**: **Gemini 1.5 / 2.0 Flash** を最優先候補とします（無料枠が非常に強力であり、有料枠でも極めて安価なため）。
* **APIキー運用 (BYOK)**: ユーザーが自身のアカウント（Google AI Studio等）で取得した個人用APIキーをアプリに入力する方式を採用し、開発者側のサーバーコストを排除します。
* **キーの安全な保持**: Windows環境のベストプラクティスに従い、**「Windows 資格情報マネージャー (Credential Manager)」**（Pythonの `keyring` ライブラリ等）を利用してOSレベルで安全に暗号化保存します。

### ⑥ 並行処理アーキテクチャ
* 音声のキャプチャ、ローカルSTT、およびLLMとの通信処理は、UIを描画するメインスレッドをブロック（フリーズ）しないよう、完全に分離された**マルチスレッド（または非同期タスク）**で実行します。
