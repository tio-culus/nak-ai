# 配信字幕・STT（Speech-to-Text）利用ツール調査報告

配信における音声認識（STT）技術は、主に「字幕表示」や「翻訳表示」、あるいは「音声合成（TTS）ツールへの入力手段」として利用されています。
本ドキュメントでは、配信コミュニティで広く使われている代表的な配信サポートツールを列挙し、その特徴と技術的な仕組み、および「STT」と「TTS」の役割の観点から調査した結果をまとめます。

---

## 1. 代表的な配信サポートツールと役割の整理

配信者のマイク音声を入力とするツールは、その目的が **「純粋な文字起こし・字幕表示（STT）」** であるか、あるいは **「別の声への変換・読み上げ（TTS）」** であるかによって、性質が大きく異なります。

### ① 音声認識字幕ちゃん (字幕ちゃん) 【純粋なSTTツール】
* **概要**: インストール不要で、Webブラウザ（Google Chrome）だけで動作するリアルタイム字幕表示ツールです。
* **目的・役割**: **純粋なSTT（音声のテキスト化・字幕表示）**。TTS（音声合成）の機能は持たず、地声で配信するストリーマーが「聞き取りやすさの向上」や「無音で視聴するリスナーへの配慮」として字幕を出すために利用します。
* **技術的な仕組み**:
  - **音声認識エンジン**: Google Chromeに標準搭載されている **Web Speech API (SpeechRecognition)** を利用。
  - **特徴**: ブラウザで完結するため配信PCへの負荷がほぼ皆無であり、日本の地声配信者における「字幕ツール」のデファクトスタンダードとなっています。

### ② ゆかりねっと / ゆかりねっとコネクター 【TTS主目的ツール（入力としてSTTを利用）】
* **概要**: 日本の配信者（特にVTuberやボイチェン配信者）の間で絶大な知名度を誇るツールです。
* **目的・役割**: **TTS（Text-to-Speech：音声合成・読み上げ）が主目的**。地声を出したくない、あるいはキャラクターの声で配信したいユーザーが利用します。
* **STTの役割**: キーボード入力の代わりに「マイク音声をリアルタイムで文字起こし（STT）し、そのテキストを即座に音声合成エンジン（VOICEROIDやVOICEVOX等）に流し込んで喋らせる」というパイプラインの中間処理としてSTTを利用しています。
* **技術的な仕組み**:
  - **音声認識エンジン**: Google Cloud Speech-to-Text API、UDトーク、ブラウザ経由の音声認識、ローカルWhisperなど、多様なSTTエンジンと連携可能です。

### ③ UDトーク (配信連携) 【高精度STTツール】
* **概要**: 本来は聴覚障害者支援や多言語コミュニケーションのためのリアルタイム文字起こしアプリですが、配信用の字幕ソースとしても広く利用されています。
* **目的・役割**: **純粋なSTT（超高精度な文字起こし）**。
* **技術的な仕組み**: 独自の高精度な音声認識プラットフォームを使用しており、固有名詞の事前辞書登録などが強力です。「ゆかりねっと」や「字幕ちゃん」の音声認識エンジンとしてバックエンドに組み込んで使われることも多いです。

### ④ OBS-Speech-to-Text (OBSプラグイン) 【配信ソフト直接組み込み型】
* **概要**: OBS Studioに直接インストールし、マイク音声ソースから直接テキスト（字幕）を生成するプラグインです。
* **目的・役割**: **純粋なSTT（字幕表示）**。
* **技術的な仕組み**: クラウドのSTT APIまたはローカルの音声認識ライブラリと連携し、OBSのテキストソースをリアルタイムに書き換えます。

## 2. Nak-AI におけるSTTアプローチへの示唆

他ツールの技術構成を分析すると、Nak-AIが「配信者の音声をリアルタイムに分析する」ためのアプローチは、主に以下の3つに分類されます。

| アプローチ | メリット | デメリット | 採用している代表ツール |
| :--- | :--- | :--- | :--- |
| **A. Web Speech API (ブラウザベース)** | - 完全無料<br>- 実装コストが極めて低い<br>- CPU/GPU負荷がほぼない | - Chrome（Chromium系）ブラウザでの起動が必須<br>- 音声認識のカスタマイズ（固有名詞の登録など）が制限される | 字幕ちゃん |
| **B. ローカル認識 (Whisper / Faster-Whisper 等)** | - クラウド通信不要でプライバシーが高い<br>- 認識精度が非常に高い（特にFaster-Whisperなど）<br>- 利用料金が一切かからない | - 高性能なPCスペック（GPU等）が必要<br>- 配信中のゲームとGPUリソースを奪い合うリスクがある | ゆかりねっと（一部） |
| **C. クラウドAPI (Google Cloud STT, Azure, Whisper API 等)** | - PCに負荷がかからない<br>- 非常に高精度で、辞書登録も可能 | - 従量課金のコストが発生する（配信のように長時間使用すると高額になるリスク） | ゆかりねっと、UDトーク、OBSプラグイン |

### Nak-AI における推奨アプローチの考察
Nak-AI は、配信者の声を別の声に変換する（TTS）必要はなく、**「地声のトーク内容をリアルタイムに分析（STT）し、配信者にアドバイスを提示する」** ソフトウェアです。

したがって、我々が技術的・体験的にベンチマークとすべきなのは、ゆかりねっとのようなTTS用途の構成ではなく、**「字幕ちゃん」や「UDトーク単体」のような、純粋なSTTをローコストかつ低遅延で実現するアプローチ**です。

* **開発プロトタイプの提案**:
  初期の検証では、最も手軽でPC負荷の低い **「A：Web Speech API（ブラウザベース）」** を用いて、音声入力の有無（無音検知）とテキスト取得の連携モデルを作るのが、素早い開発に適していると考えられます。
